# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

The **Libre Listener Wallet** is a zero-infrastructure, non-custodial Bitcoin Lightning wallet that runs inside browser/PWA sandboxes (and native mobile wrappers) for the Podcasting 2.0 / Value-for-Value (`v4vmusic.com`) ecosystem. It wraps **LDK (Lightning Development Kit) WASM**, persists node state in IndexedDB, opens just-in-time channels via LSP protocols, and sends V4V "boost" payments as keysend with bLIP-10 TLV metadata.

> Experimental: not production-ready. The README warns loss of funds is likely.

## Architecture

TypeScript monorepo (`packages/*`). Cross-package imports go through package **names**, never relative paths.

- **`packages/shared`** (`@libre/shared`) — single source of truth for protocol types: `WalletConfig`, NWC request/response, LSPS1/LSPS2 JSON-RPC shapes, V4V utils, Zod schemas. Anything shared between SDK and gateway lives here; do not duplicate type signatures across packages.
- **`packages/libre-listener-wallet`** (`@libre/listener-wallet`) — the client SDK. The `LibreListenerWallet` class (`src/index.ts`) adapts LDK's low-level bindings into a simple API: `start()`, `stop()`, `connectPeer()`, `createInvoice()` (one shared private `buildInvoice()` behind it, `requestLSPS2Invoice()`, and NWC `make_invoice`), `sendKeysendPayment()` (the single keysend path — NWC `pay_keysend` routes through it), `syncGossip()`, `onStateChanged(cb)` (fires when channel state is persisted), read-only `getChannels(): ChannelInfo[]` / `getBalance(): { spendableSat, receivableSat }` (mapped from LDK `list_channels()` via the pure `mapChannelDetails`/`sumBalance`; summed over **usable** channels), and encrypted backup/recovery: `exportState({ passphrase? })`, `importState(envelope, secret)` (secret = passphrase or 64-hex seed; auto-detects format; enforces network match), `verifyBackup(envelope, secret)` (decrypts WITHOUT writing storage — used to prove a backup is restorable before funding; returns `{ ok, hasSeed, seedMatches, network, entryKeys }`), and **force-close sweep**: `setSweepDestination(scriptPubKey?)` sets where reclaimed funds go, and on `Event_SpendableOutputs` (a channel close) the SDK sweeps the claimable on-chain outputs there via `PhantomKeysManager.as_OutputSpender().spend_spendable_outputs(...)` → esplora broadcast (built synchronously in the event so the descriptors stay valid; only the tx bytes broadcast async). If no sweep address is set (or the build fails) the handler returns `Err(ReplayEvent)` so LDK re-delivers the event until it's actually swept — funds are never dropped (proven on mainnet recovering a force-closed `to_remote`). Supporting modules: `nwc-manager` (takes `{logger,storage,network}` via its constructor — no reaching into wallet internals; emits **NIP-47 `payment_sent`/`payment_received` notifications** (kind 23196, advertised in `get_info` + the 13194 info event) so apps reconcile a settlement that landed after their request timed out, and **caps the synchronous settlement-await at 90s** so a stuck HTLC can't freeze a client's request chain — see `awaitWithTimeout`), `lsps-client`, `esplora-client`, `crypto-utils`, `state-backup` (backup envelope crypto — see below), `indexed-db-storage`, `storage-cache`, `peer-reconnect` (exponential-backoff delay calc). **Peer auto-reconnect:** `connectPeer` records "desired" peers and `handleDisconnect` redials them with exponential backoff (1s→2s→…cap 30s) while the node runs (`stop()` cancels pending redials) — a dropped peer otherwise silently disables sending ("0 first hops"). `WalletConfig` extras: `announceChannels`, `alias` (broadcast a node_announcement), `trustedZeroConfPeers`, `rapidGossipSyncUrl`, `autoReconnectPeers` (default on; set false to disable). Built to CJS+ESM via `tsup`.
- **`packages/libre-nwc-push-gateway`** (`@libre/nwc-push-gateway`) — stateless Express server (`LibreNWCPushGateway`) that subscribes to Nostr relays and sends Web Push to wake offline PWAs for NWC/NIP-47 requests. Also exposes a **CORS-enabled RGS passthrough** (`GET /rgs/snapshot/:timestamp`) that fetches the LDK gossip snapshot server-side and re-serves it with CORS, so browsers can populate the network graph for multi-hop routing (see the RGS gotcha below). SQLite-backed (`better-sqlite3`), CJS build. Deployed on Railway (`nwc-push-gateway-production.up.railway.app`).
- **`packages/libre-lsps2-server`** (`@libre/lsps2-server`) — a **regtest-only dev LSP** that funds the listener through the app's existing LSPS2 flow (`lsps2.get_versions`/`get_info`/`buy`). On `buy` it opens a real channel from `libre-lnd` to the client (`LndRestClient.openChannel`, **non-anchor** `STATIC_REMOTE_KEY` to dodge the regtest anchor-fee mismatch), mines to confirm (`BitcoindClient.mineBlocks` to a **fixed** address — `getnewaddress` needs a loaded wallet), and returns the real confirmed SCID (`findChannelScid` matches by `remote_pubkey` — lnd's `OpenChannelSync` returns `funding_txid_bytes`, not `_str`). Express on `127.0.0.1:9099` (`POST /lsps2`), config via env (`server.cjs` dev entry sets `NODE_TLS_REJECT_UNAUTHORIZED=0` for lnd's self-signed cert). Point the app's **LSP API URL** at it (the default) and click **Request Invoice** to fund. Not real HTLC-interception JIT; the BOLT11 is cosmetic. **Funding → send → receive is proven end-to-end on regtest.**
- **`packages/example-app`** (`@libre/example-app`) — Vite PWA playground. `main.ts` is being slimmed into a bootstrap that wires feature modules (`web-push.ts`, `nwc-ui.ts`, `v4v.ts`, …) plus `core/` (`logger.ts`, `ui-helpers.ts`, `app-context.ts`). Pattern: mutable state (`wallet`, `isNodeRunning`) stays in `main.ts`; modules receive **live accessors** via `AppContext` (`getWallet()`, `isRunning()`) passed to `initX(ctx)`, and expose `setXEnabled()` that the start/stop handlers call (so node-control stays decoupled). The remaining handlers (node start/stop, peer-connect, lsp-capacity, receive, backup/drive, network-config, seed) are still inline in `main.ts` — extract them the same way. `core/persistent-storage.ts` (`ensurePersistentStorage()`) is the one app-layer module that may touch `navigator`. The **mainnet preset + Google Client ID read from a gitignored `.env.local`** (`VITE_MAINNET_PEER`, `VITE_MAINNET_BRIDGE`, `VITE_GOOGLE_CLIENT_ID`, `VITE_MAINNET_RGS` = the gateway `/rgs/snapshot` URL). `tsup.config.ts` bundles `src/service-worker.ts` into `public/` separately from the Vite build. Has its own vitest setup (`pnpm --filter @libre/example-app test`). **Page layout (`index.html`):** a focused top **`.setup-zone`** (the only thing you set up first) — a full-width **status strip** (`.stat-grid`: node id, peers, spendable/receivable, channels, routing graph) over three `.card` panels **Wallet & Seed / Node / Backup & Recovery** — sits above the **"Protocol & Feature Tests"** grid (`.dashboard-grid`) holding the rest (LSP, LSPS2, LSPS1, V4V audio, Receive, Boost, NWC, Web Push, Console). All handlers bind by element `id`, so the HTML can be restructured without touching JS. Cards have no hover-lift (it shuffled the grid). The status strip is the live **balance + channel-status view** (`refreshWalletView()` reads `getBalance()`/`getChannels()`, refreshed on start/`onStateChanged`/LDK-events + a 5s poll, cleared on stop) with per-channel **active/ready/pending** badges (`isUsable`/`isChannelReady`). The LSP card has an **LSP preset selector** (Custom/dev · Megalith mainnet) that fills the verified peer connection, and the LSPS1 card a **channel lease-duration picker** wired to `purchaseLSPS1Capacity`'s `channelExpiryBlocks` (clamped to the LSP's advertised `[min,max]`, defaults to the MAX — longest channel life per open fee; `announceChannel` defaults false since a browser node has no reachable address). The app passes **`trustedZeroConfPeers: []`** — it accepts normal confirmed channels; 0-conf-accepting a non-0-conf open made lnd reject it (`min depth zero`), and no current LSP (the dev server opens confirmed channels; Mutinynet doesn't signal zeroconf) needs 0-conf.

  **Onboarding & backup flow (seed-only):** **Create New Wallet** reveals a freshly generated seed (no hardcoded default) → user ticks "I've saved my recovery seed" → **Create Wallet** wipes storage, writes the seed, and auto-starts the node, which best-effort syncs + verifies the encrypted backup to Drive. Start refuses a generated-but-not-created wallet (no bypass). `ensurePersistentStorage()` **warns** (does not block) when storage isn't durable — `navigator.storage.persist()` returns false on ordinary localhost windows, so a hard block was a false positive; the saved seed + verified backup are the real safety net. Auto-start-on-load is default-on (persisted checkbox). Drive backups are **seed-encrypted** and auto-sync (5s-debounced on every state-version advance, with a catch-up sync on (re)connect — kept deliberately frequent; restorability is prioritized over Drive-write count). Restore is seed-only, and **auto-detects the network** from the Drive backup(s) (`drive-backup.ts` `listBackupNetworks` lists `libre-wallet-backup-*.json` → `pickRestoreNetwork` prefers mainnet) so the user doesn't pre-select a network; default network is mainnet. **Start-readiness guard (`core/wallet-readiness.ts` `assessStartReadiness`):** a seed with no `channel_manager` may auto-start/connect ONLY if it was created brand-new here (provenance marker `wallet_created_new`, set on Create Wallet, cleared once channel state persists) — otherwise auto-start AND manual Start are blocked with a restore prompt. This prevents a seed-without-state (e.g. an incomplete restore) from bootstrapping a fresh empty node, connecting the peer, and replying "unknown channel" to its `channel_reestablish` → which **force-closes the channel** (see gotcha below). **On-chain sweep address field** (`core/address-script.ts` `addressToScriptPubKey` decodes bech32/bech32m via `@scure/base`) is persisted (`libre_sweep_address`) and pushed to `wallet.setSweepDestination` on edit + on start, so a force-close auto-recovers funds. **Drive reconnect:** the access token is in-memory only, so each reload re-auths; the app remembers the account email (`email` scope) and feeds it as a `login_hint` for a silent reconnect, and because Google's token flow needs a user gesture (a page-load attempt fails `popup_failed_to_open`), it retries the silent reconnect on the user's **first interaction** (`core/drive-ui.ts` `shouldArmGestureReconnect`). The **Connect Drive button** turns green "Connected ✓" via `driveButtonView`. (The SDK retains the optional passphrase/dual-wrap capability; the app doesn't surface it. BIP39 word seeds are a planned follow-up.)

### Core design patterns (enforced)

- **Dependency injection / platform abstraction:** The SDK must NOT import platform modules (`fs`, `window`, React Native `SecureStore`) directly. Storage, WebSocket transport, and logging are injected as interfaces (`SecureStorageProvider`, `WebSocketStreamProvider`, `Logger`) via the wallet constructor. Web injects IndexedDB; mobile injects Keychain.
- **No stateful singletons:** Never `export const node = new LDKNode()`. Instantiate the node inside a class instance so tests can run concurrently and re-init cleanly.
- **No hardcoded env in the SDK:** All config (network, Esplora endpoint, whitelisted relays/LSPs) is passed into the constructor — no `process.env` reads inside the library.
- **Barrel exports only:** Import from package `index.ts`, not deep relative paths.
- **Errors:** No silent catches. Log via the injected `Logger` and rethrow or return `{ ok: false, error: string }`.

### V4V / boostagram TLV (keysend)

- Key `7629169`: UTF-8 JSON `BoostRecord` (bLIP-10) — **not** hex-encoded.
- Key `7629175`: Podcast Index `feedGuid` as a plain string.
- `boost_uuid` is shared across all split recipients; `uuid` is unique per-recipient. See `project-conventions.md` for the full payload schema.

### Backup format (`state-backup.ts`)

- **v2 (current) = envelope encryption.** A random 32-byte DEK encrypts the `BackupPayload` (AES-256-GCM); the DEK is wrapped to **both** a passphrase (PBKDF2-SHA256, 600k iters, random salt) and the seed (HKDF-SHA256, info `libre-wallet-backup-kek-v2`), so **either secret independently decrypts** the backup. `serializeAndEncrypt(payload, { passphrase, seedHex })` emits v2.
- **v1 (legacy) = seed-only** (HKDF info `libre-wallet-backup-v1`, AES-GCM). `serializeAndEncryptV1(payload, seedHex)` still exists; `decryptAndParse(envelope, secret)` auto-detects v1/v2. `exportState()` with no passphrase emits v1 (what the example app uses today).
- `BackupPayload.entries` holds `ldk_seed` + `channel_manager` + monitor keys + `network_graph`/`scorer`/`state_version` — i.e. the seed is *inside* the (encrypted) backup. The top-level (non-monitor) key set is pinned as `BACKUP_DIRECT_KEYS` (`backup-keys.ts`); `exportState` appends the monitor keys from `ldk_keys_index`. All WebCrypto; no deps. The DB-not-namespaced-by-network gotcha means a backup's `network` is checked on import.

## Critical Guardrails

- **Absolute key isolation:** Seed phrases, private keys, and unclaimed-HTLC preimages must never leave the client sandbox — never over sockets, HTTP, or logs. Backups must be encrypted locally first.
- **Zero-custody gateway:** The push gateway must never hold node keys or NWC shared secrets; it routes blind, encrypted Nostr envelopes only.
- **DB isolation:** The gateway uses its own standalone SQLite/Postgres DB — never the host app's database.
- **Zero-conf LSP vetting:** Accept 0-conf JIT channels only from trusted LSPs. Enforced on the acceptance side via `WalletConfig.trustedZeroConfPeers` (allowlist of counterparty pubkeys); non-allowlisted peers fall through to a normal, confirmation-gated channel. Never 0-conf from random gossip nodes.
- **Localhost binding:** All docker/testing services map ports to `127.0.0.1` only.

## Known limitations & gotchas (hard-won — read before debugging "it broke")

- **Wallet storage IS namespaced by network (fixed).** Each network uses its own IndexedDB (`libre-wallet-<network>`); the example app builds a network-scoped `IndexedDBStorageProvider` (`core/storage-namespace.ts` → `dbNameForNetwork`), auto-migrates the legacy un-namespaced `libre-wallet` DB into the right network DB on first load (`migrateStorage`, idempotent, copies `ldk_seed` last for crash-safety, legacy DB left intact), and disables the network selector while the node runs. Off-page consumers (service worker, web-push simulate) read the active network from a fixed `libre-wallet-meta` pointer (`ACTIVE_NETWORK_KEY`). Historically all networks shared one DB and switching profiles corrupted channel state (`Loaded 0 channel monitors` + a wrong-chain manager) — that's resolved. Each network is an independent wallet (own seed).
- **The example app is deployed to GitHub Pages, and a storage-layer change can orphan a funded wallet.** `.github/workflows/deploy-pages.yml` builds `@libre/example-app` and publishes to `https://chadfarrow.github.io/libre-listener-wallet-monorepo/` on **every push to `master`** that touches `packages/{example-app,shared,libre-listener-wallet}/**` (native Actions Pages pipeline; `vite.config.ts` `base: "./"`; mainnet `VITE_*` come from repo *Variables*). Wallet state lives in that origin's IndexedDB independent of the code, so UI/feature pushes are safe — but renaming a persisted key, changing a DB name, or altering the backup format makes the node unable to find its channel state on next load (**force-close**) or makes existing encrypted backups undecryptable. **Guard (defense-in-depth):** the **storage-contract tests** (`packages/libre-listener-wallet/src/tests/unit/storage-contract.test.ts` + `packages/example-app/src/core/storage-contract.test.ts`) pin every on-disk invariant — DB name format / object store `settings` / version `1`, the `libre-wallet-meta`+`active_network` pointer, the Drive filename, and the backup envelope format via **frozen golden v1/v2 envelopes that must keep decrypting**. `pnpm check:storage` runs just these (unit, no docker/WASM). A **husky `.husky/pre-push`** hook hard-blocks a push when an invariant changed (override: `git push --no-verify`) and warns when any storage-layer file is touched; the **deploy workflow re-runs `check:storage` before build** so a `--no-verify` push (or a clone without the hook) still can't ship a broken invariant — the build fails and production stays on the last good version. When you intentionally change a storage constant, add a migration and update the contract test (incl. regenerating golden envelopes) in the **same commit**.
- **Rapid Gossip Sync is CORS-blocked at the source — solved via a proxy.** `rapidsync.lightningdevkit.org` sends no `Access-Control-Allow-Origin`, so a direct browser `fetch` fails (it only works from Node tests). Without the graph there's **no multi-hop routing** (e.g. a v4vmusic boost to an arbitrary artist fails to find a route). The fix: the **push gateway exposes a CORS-enabled passthrough at `GET /rgs/snapshot/:timestamp`** (`packages/libre-nwc-push-gateway/src/index.ts`) that fetches the LDK snapshot server-side and re-serves it with CORS. The example app points `rapidGossipSyncUrl` at it via **`VITE_MAINNET_RGS`** (resolved mainnet-only by `core/rgs-config.ts`; the LDK RGS server only serves mainnet snapshots), warms the graph on node start, and exposes a manual **Sync gossip** button + a graph-channel-count readout. Live P2P gossip is still disabled (`IgnoringMessageHandler`). Caveat: multi-hop only succeeds if the wallet's channel peer has a **public/announced** path onward (the RGS graph only contains announced channels) — a peer whose only channel is private to the wallet is a routing dead-end.
- **Public Esplora endpoints rate-limit.** `mempool.space` → "Failed to fetch" (CORS/Cloudflare), `blockstream.info` → `429 Too Many Requests` once a Start's burst of block-by-block sync requests (compounded by repeated Starts) trips the limit. The example app now defaults mainnet to `mempool.space/api`; don't spam Start. **Auto-start-on-load is enabled by default** (a wallet should just run) but exposed as an **"Auto-start node on load" checkbox** (persisted in `localStorage` as `libre_autostart`); untick it if public-Esplora rate limits bite on repeated reloads.
- **A browser node has no listening socket** — nothing can dial it. It connects *out* through a **websockify bridge** to a peer, and the peer opens the channel back over that existing connection. Tor-only peers need a `tor` + `socat` (SOCKS4A) + `websockify` chain.
- **Esplora txid byte-order (fixed — was why channels never confirmed in-browser).** LDK hands txids as raw bytes in **little-endian** (internal) order; esplora's REST API addresses txs by **big-endian display** hex (the reverse). `esplora-client.ts` must convert via `ldkTxidToDisplay()` before any esplora lookup (`register_tx`, `register_output`, and the `get_relevant_txids()` query sites) — but **keep little-endian when feeding a txid back to LDK** (e.g. reorg `transaction_unconfirmed`). Symptom of the bug: funding tx confirmed on-chain + lnd sees the channel, but the wallet stays `pending` forever (`isChannelReady=false`), never sends `channel_ready`, so lnd's link stays `active:false` and payments fail `INSUFFICIENT_BALANCE`.
- **Buried-confirmation catch-up (fixed).** `sync()`'s forward loop only confirms txs whose block is in `(bestHeight, tip]`. A funding tx registered **after** its block was already synced — instant regtest mining, or app closed → funding confirms → reopen past the block (a real **mainnet PWA restart race**) — would never confirm. `sync()` runs a catch-up pass (`planBuriedConfirmations`) that confirms watched/registered txs buried at height `≤ bestHeight` at their real height. `sync()` runs every 30s, so stuck channels recover on the next tick.
- **A seed without channel state must never auto-start/connect (force-close risk — fixed).** A bare seed with no `channel_manager` (e.g. a restore that didn't complete, or a seed injected into a fresh origin) that auto-starts will bootstrap a fresh empty ChannelManager, connect the peer, and on the peer's `channel_reestablish` reply "unknown channel" → **force-close the real channel** (this actually happened on mainnet during the web migration). Fix: `example-app/src/core/wallet-readiness.ts` `assessStartReadiness` gates auto-start AND manual Start — a seed with no channel state may start only if created brand-new here (`wallet_created_new` provenance marker). The **force-close sweeper** (`setSweepDestination` + `Event_SpendableOutputs`) is the recovery net if it ever happens; a `StaticRemoteKey` channel's `to_remote` is seed-claimable.
- **The service worker must BUNDLE every dependency (no bare specifiers).** `src/service-worker.ts` is compiled by **tsup** (invoked from the vite `copyLdkWasmPlugin` `buildStart`, so `vite build` — and thus CI/Pages deploy — builds it too) and bundles the full SDK. tsup **auto-externalizes `package.json` dependencies**, so any dep the SW transitively needs must be added to tsup's `noExternal` — otherwise a bare `import … from "x"` survives in the bundle and the browser SW dies with **"ServiceWorker script evaluation failed"** (a browser ServiceWorker can't resolve bare specifiers). This bit us when `@scure/base` was added (pulled in transitively via `nostr-tools`): the fix was adding it to `noExternal`. Debug by evaluating `public/service-worker.js` in a SW-scope sandbox (define `self`/`caches`/`clients`, leave `window`/`document` undefined) — Node resolves bare specifiers so it passes; the browser doesn't.

## Testing Rules (Vitest)

- **TDD is mandatory:** red → green → refactor. No production code changes without corresponding tests in the same task.
- **Do NOT mock LDK internals** (`vi.mock('lightningdevkit')`, channel managers). Use the real LDK WASM library; mock the network socket transport or HTTP via **MSW** instead.
- **Do NOT mock the gateway DB.** Use a real isolated test DB (in-memory SQLite); truncate between suites.
- **Assert outcomes, not implementation** (node status transitions, persisted state, relay responses) — not private call order or log strings.
- Wallet tests run in `jsdom` (IndexedDB simulation); gateway tests run in `node`.
- Integration tests in `packages/libre-listener-wallet/src/tests/integration/` require the `docker compose` regtest stack (esplora on `:3002`, lnd LSP on `:9735`). Unit tests in `src/tests/unit/` do not.
- **Storage-contract tests are special:** the `storage-contract.test.ts` files pin the on-disk format that funded wallets depend on. Never "fix" a failing assertion by editing the expected value — a failure means a persisted invariant changed (force-close / unrestorable-backup risk). Change the constant deliberately + add a migration + update the test in the same commit. They're wired into `pnpm check:storage`, the pre-push hook, and the deploy workflow (see the storage-contract guard above).
